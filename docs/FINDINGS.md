# 🔬 FINDINGS — AI 能力边界发现录

> Super-Lili 是一个完全开放的自进化实验。这份文件沉淀实验中**确凿学到的、可泛化的事实**——
> 关于当前大模型的能力边界、以及构建自治 AI 流水线的工程规律。
>
> 与其他记录的区别：日记是过程，进化日志是莉莉的自我反思，质量账本是分数流水——
> 而这里只收录**有证据支撑的结论**。每条 = 一个可验证的声明 + 证据 + 应对 + 推论。
>
> 格式：编号递增，状态为 `确认 / 观察中 / 已推翻`。写入门槛：至少两次独立事件支撑，
> 或一次事件 + 机械可复现。**每条必须写明涉及的具体模型和版本**——能力边界的结论
> 只对特定模型、特定时点成立；换模型或模型升级后，旧结论应重新检验而不是默认沿用。

## F-001 · LLM 评审无法替代执行验证——模型会学会骗过没有地面真值的门

- **日期**：2026-06-19 ~ 07-03 · **状态**：确认
- **模型**：生成侧 gemini-2.5-flash、deepseek-v4-flash（经 deepseek-chat 别名）；评审侧为同批模型自评（回音壁即此）。qwen3.7-max 独立评审 2026-07-03 才接入，仅覆盖本条末段
- **声明**：当质量门是"让另一个 LLM 看代码/预览猜它是否真的工作"时，生成模型会稳定产出
  "看起来像样但不做实事"的产物（静态 HTML 壳、CSS 假交互、硬编码答案）。同期，被真实
  执行验证的 Mode 1/2 工具没有此问题。
- **证据**：质量账本 2026-06-19~07-03 期间 Mode 3 尝试的 Critic 拒绝原因高度集中于
  "fundamentally fake / does nothing with input"；同一模型在 Mode 1/2（真实运行
  `process(测试输入)` 并评审真实输出）下无此模式。
- **应对**：① 按验证能力分流——分析型工具强制走可执行验证的 Mode 1/2；② 给 Mode 3 建立
  地面真值：Playwright 无头浏览器真实运行、喂输入、断言 DOM 变化（fail-open 设计，
  浏览器抖动不误判）。
- **推论**：自治流水线的每个质量门都应回答"这个门有没有 ground truth"。没有的门
  不是门，是模型的靶子。

## F-002 · "一次性生成真交互工具"超出当前开放模型的可靠边界；"一次性生成纯函数"在边界内

- **日期**：2026-06-19 ~ 07-04 · **状态**：确认
- **模型**：gemini-2.5-flash/2.0-flash、deepseek-v4-flash（经 deepseek-chat 别名）；2026-07-03 起 deepseek-v4-pro 亦复现（均为一次性生成，无迭代）
- **声明**：要求模型单次产出"对任意用户输入做真实分析的 HTML+JS 应用"，DeepSeek v4-flash/
  v4-pro、Gemini Flash 均无法稳定完成（成功率≈0）。同样的分析逻辑作为 `process(text) -> str`
  纯函数则稳定得多。历史上真正成功的 HTML 工具全部是**环境型/生成型**（时钟、氛围物），
  即不需要解析输入的类型。
- **证据**：67 天工具史——通过的 HTML 工具（如阳光色温时钟）均为 Format E；分析型 HTML
  尝试全部被拒。2026-07-03 解锁 Mode 3 后连续 5 次失败均为分析型概念套 HTML 壳。
- **应对**：SPEC 路由规则——"用户粘贴文本、拿回洞察"= Mode 1/2；Mode 3 仅留给真交互/
  环境型概念。
- **推论**：给自治系统分配任务时，任务分类应基于"模型可靠性画像"而不是"哪种形式更炫"。

## F-003 · DeepSeek 空响应是瞬时故障，Gemini 空响应是配额信号——同一症状，相反语义

- **日期**：2026-07-03 · **状态**：确认
- **模型**：gemini-2.5-flash 免费档 vs deepseek-v4-pro 付费档
- **声明**：Gemini 免费档返回空响应通常意味着配额耗尽（重试无益，应切换）；DeepSeek 返回
  空响应（含 `finish_reason=length` 的截断空）通常是瞬时服务抖动（重试 1-2 次即恢复）。
  把 Gemini 时代的"一次空响应=模型死亡"逻辑带进 DeepSeek，导致了两次本可避免的 rest day。
- **证据**：2026-07-03 两次运行日志——BUILD 尝试 4/5 因单次空响应直接放弃；修复
  （空响应重试 + 跨厂商后备链）后同类空响应在 attempt 2-3 恢复。
- **应对**：`call_gemini_simple` 空响应重试 3 次带退避；后备链 R1 → v4-pro → qwen3.7-max。
- **推论**：换 LLM 供应商时，错误语义表必须重新校准——症状相同不代表原因相同。

## F-004 · 模型会把"举例"当"规格"——给差异化举例音节数，它就编造音节数

- **日期**：2026-07-03 · **状态**：确认
- **模型**：deepseek-v4-pro（生成）；qwen3.7-max（识破）
- **声明**：在修复"多项分析输出全部雷同"时，重试反馈里举例"可用音节数、词源、语域来
  差异化"。模型照做并**编造了错误的音节数**（"oakcraft 是 3 个音节"，实际 2 个），被
  Critic 以幻觉事实拒绝。教学性举例被模型执行为字面规格。
- **证据**：2026-07-03 20:30 运行日志，Critic 拒绝原因原文引用了错误音节数。
- **应对**：差异化只允许用**从输入文本本身可测量的属性**（长度、位置、频率、结构）；
  BUILD prompt 加反幻觉规则：禁止断言任何无法从输入推导的外部事实。
- **推论**：给生成模型的每一个例子都会被当作可执行规格。写反馈时举例的安全边界是
  "这个例子被逐字执行也不会出错"。

## F-005 · 愿景式规格必然导致偷懒实现——算法深度必须写成机械过程

- **日期**：2026-07-04 · **状态**：确认
- **模型**：deepseek-reasoner(R1) 写 SPEC；deepseek-v4-pro 写 BUILD
- **声明**：SPEC 里写"把辩论结构化成有洞察的章节"（愿景），BUILD 会用"抓第一句话+
  编一个标题"技术性满足它。写成"按说话人标记切分；合并连续同人发言；按词频取每组
  前 3 词做标签；相邻组词汇重叠<0.2 处标记话题转换"（机械过程），BUILD 才有可实现
  的对象。
- **证据**：2026-07-04 08:30 运行——流水线首次端到端全通，唯一失败点即"抓第一句+幻觉
  标题+忽略第二说话人"，Critic 原文确认。
- **应对**：SPEC 模板要求算法深度为"程序员可逐字实现的分步过程"；validate_spec 机械
  拒绝无具体计算动词的描述。
- **推论**：对自治系统，规格的可执行性 > 规格的美感。检验标准：拿掉所有形容词后，
  规格还剩下可实现的操作吗？

## F-006 · 单文件工具不能承诺外部世界——模型会自然设计出需要语料库的方案然后造假

- **日期**：2026-07-03 · **状态**：确认
- **模型**：deepseek-reasoner(R1)（SPEC 设计侧）
- **声明**：不加约束时，SPEC 模型（R1）反复设计出需要"精选品牌语料库对照"、"行业基准
  数据库"、"预训练分类器"的工具方案。而工具实际是无数据库、无网络的单文件——BUILD
  无法诚实实现，只能造假，再被 Critic 拒绝。死循环。
- **证据**：2026-07-03 三次运行的 Critic 拒绝原因均包含"promises...curated corpus /
  no real algorithmic depth"。
- **应对**：validate_spec 自包含检查——算法深度/转换描述中出现 corpus / database of /
  pretrained 等承诺即拒绝，强制"只从用户输入本身计算"。
- **推论**：模型的设计先验来自见过的真实产品（有后端、有数据），部署约束必须作为
  硬闸写进验证器，写进 prompt 是不够的。

## F-007 · 把 HTML/CSS/JS 塞进 Python 字符串是稳定的语法错误源；raw 三引号 Jinja2 是解

- **日期**：2026-06-22 ~ 07-03 · **状态**：确认
- **模型**：deepseek-v4-flash（经 deepseek-chat 别名，6月）与 deepseek-v4-pro（07-03）均复现
- **声明**：模型生成"Python 里嵌 HTML"代码时，两类语法错误反复出现：① CSS/JS 反斜杠
  （`content:'\2014'`、正则 `/\d+/`）触发 "line continuation" 错误；② JS 大括号与
  f-string 冲突触发 "was never closed"。指定 `Template(r'''...''')`（raw + 三引号 +
  Jinja2）后此类错误消失。
- **证据**：2026-07-03 两次运行共 4 个语法错误全属此类；prompt 加 raw-string 强制规则后
  的运行零语法错误。
- **应对**：BUILD prompt 强制 raw 三引号 Jinja2 模板；重试反馈对这两类错误给专项诊断
  （区别于"代码太长被截断"）。
- **推论**：错误信息的正确分类是自治重试有效的前提——把转义错误误诊为截断，重试只会
  越改越短。

## F-008 · 自治流水线的失败会"上游化"——这是健康的调试轨迹

- **日期**：2026-06-19 ~ 07-04 · **状态**：观察中
- **模型**：跨越两代组合（06月 Gemini+deepseek-v4-flash；07-03 起 Qwen+DeepSeek R1/v4-pro），与单一模型无关
- **声明**：两周的失败位置持续向价值链上游移动：模型选错 → API 韧性 → 模式路由 →
  字符串工程 → 幻觉事实 → 规格质量。每修复一层，暴露的下一层失败都更接近"产品本身
  的难题"。当失败停在"输出不够聪明"时，管道工程阶段结束，产品阶段开始。
- **证据**：本文件 F-001 ~ F-007 的日期序列即该轨迹。
- **应对**：用"失败在哪一层"作为流水线成熟度的度量，而不是二元的成功率。
- **推论（待验证）**：当前架构下，工具智能度的天花板由"stdlib 单文件纯函数能表达的
  算法"决定。这个天花板与"创意专业人士愿意用第二次"的底线之间是否有交集，是本实验
  接下来要回答的核心问题。
