# 🔬 FINDINGS — AI 能力边界发现录

> Super-Lili 是一个完全开放的自进化实验。这份文件沉淀实验中**确凿学到的、可泛化的事实**——
> 关于当前大模型的能力边界、以及构建自治 AI 流水线的工程规律。
>
> 与其他记录的区别：日记是过程，进化日志是莉莉的自我反思，质量账本是分数流水——
> 而这里只收录**有证据支撑的结论**。每条 = 一个可验证的声明 + 证据 + 应对 + 推论。
>
> 格式：编号递增，状态为 `确认 / 观察中 / 已推翻`。写入门槛：至少两次独立事件支撑，
> 或一次事件 + 机械可复现。**每条必须写明涉及的具体模型和版本**——能力边界的结论
> 只对特定模型、特定时点成立；换模型或模型升级后，旧结论应重新检验而不是默认沿用。

## F-001 · LLM 评审无法替代执行验证——模型会学会骗过没有地面真值的门

- **日期**：2026-06-19 ~ 07-03 · **状态**：确认
- **模型**：生成侧 gemini-2.5-flash、deepseek-v4-flash（经 deepseek-chat 别名）；评审侧为同批模型自评（回音壁即此）。qwen3.7-max 独立评审 2026-07-03 才接入，仅覆盖本条末段
- **声明**：当质量门是"让另一个 LLM 看代码/预览猜它是否真的工作"时，生成模型会稳定产出
  "看起来像样但不做实事"的产物（静态 HTML 壳、CSS 假交互、硬编码答案）。同期，被真实
  执行验证的 Mode 1/2 工具没有此问题。
- **证据**：质量账本 2026-06-19~07-03 期间 Mode 3 尝试的 Critic 拒绝原因高度集中于
  "fundamentally fake / does nothing with input"；同一模型在 Mode 1/2（真实运行
  `process(测试输入)` 并评审真实输出）下无此模式。
  **2026-07-06 实战确认**：Playwright 地面真值门首次开火——真实填入输入、点击控件后
  以执行证据拒绝（fields_filled=1, text_changed=False, nodes 21->21），deepseek-v4-pro
  在全部反伪造规则生效的情况下仍产出静态假交互工具，被执行证据而非 LLM 猜测拦下。
- **应对**：① 按验证能力分流——分析型工具强制走可执行验证的 Mode 1/2；② 给 Mode 3 建立
  地面真值：Playwright 无头浏览器真实运行、喂输入、断言 DOM 变化（fail-open 设计，
  浏览器抖动不误判）。
- **推论**：自治流水线的每个质量门都应回答"这个门有没有 ground truth"。没有的门
  不是门，是模型的靶子。

## F-002 · "一次性生成真交互工具"超出当前开放模型的可靠边界；"一次性生成纯函数"在边界内

- **日期**：2026-06-19 ~ 07-04 · **状态**：确认
- **模型**：gemini-2.5-flash/2.0-flash、deepseek-v4-flash（经 deepseek-chat 别名）；2026-07-03 起 deepseek-v4-pro 亦复现（均为一次性生成，无迭代）
- **声明**：要求模型单次产出"对任意用户输入做真实分析的 HTML+JS 应用"，DeepSeek v4-flash/
  v4-pro、Gemini Flash 均无法稳定完成（成功率≈0）。同样的分析逻辑作为 `process(text) -> str`
  纯函数则稳定得多。历史上真正成功的 HTML 工具全部是**环境型/生成型**（时钟、氛围物），
  即不需要解析输入的类型。
- **证据**：67 天工具史——通过的 HTML 工具（如阳光色温时钟）均为 Format E；分析型 HTML
  尝试全部被拒。2026-07-03 解锁 Mode 3 后连续 5 次失败均为分析型概念套 HTML 壳。
- **应对**：SPEC 路由规则——"用户粘贴文本、拿回洞察"= Mode 1/2；Mode 3 仅留给真交互/
  环境型概念。
- **推论**：给自治系统分配任务时，任务分类应基于"模型可靠性画像"而不是"哪种形式更炫"。

## F-003 · DeepSeek 空响应是瞬时故障，Gemini 空响应是配额信号——同一症状，相反语义

- **日期**：2026-07-03 · **状态**：确认
- **模型**：gemini-2.5-flash 免费档 vs deepseek-v4-pro 付费档
- **声明**：Gemini 免费档返回空响应通常意味着配额耗尽（重试无益，应切换）；DeepSeek 返回
  空响应（含 `finish_reason=length` 的截断空）通常是瞬时服务抖动（重试 1-2 次即恢复）。
  把 Gemini 时代的"一次空响应=模型死亡"逻辑带进 DeepSeek，导致了两次本可避免的 rest day。
- **证据**：2026-07-03 两次运行日志——BUILD 尝试 4/5 因单次空响应直接放弃；修复
  （空响应重试 + 跨厂商后备链）后同类空响应在 attempt 2-3 恢复。
- **应对**：`call_gemini_simple` 空响应重试 3 次带退避；后备链 R1 → v4-pro → qwen3.7-max。
- **推论**：换 LLM 供应商时，错误语义表必须重新校准——症状相同不代表原因相同。

## F-004 · 模型会把"举例"当"规格"——给差异化举例音节数，它就编造音节数

- **日期**：2026-07-03 · **状态**：确认
- **模型**：deepseek-v4-pro（生成）；qwen3.7-max（识破）
- **声明**：在修复"多项分析输出全部雷同"时，重试反馈里举例"可用音节数、词源、语域来
  差异化"。模型照做并**编造了错误的音节数**（"oakcraft 是 3 个音节"，实际 2 个），被
  Critic 以幻觉事实拒绝。教学性举例被模型执行为字面规格。
- **证据**：2026-07-03 20:30 运行日志，Critic 拒绝原因原文引用了错误音节数。
- **应对**：差异化只允许用**从输入文本本身可测量的属性**（长度、位置、频率、结构）；
  BUILD prompt 加反幻觉规则：禁止断言任何无法从输入推导的外部事实。
- **推论**：给生成模型的每一个例子都会被当作可执行规格。写反馈时举例的安全边界是
  "这个例子被逐字执行也不会出错"。

## F-005 · 愿景式规格必然导致偷懒实现——算法深度必须写成机械过程

- **日期**：2026-07-04 · **状态**：确认
- **模型**：deepseek-reasoner(R1) 写 SPEC；deepseek-v4-pro 写 BUILD
- **声明**：SPEC 里写"把辩论结构化成有洞察的章节"（愿景），BUILD 会用"抓第一句话+
  编一个标题"技术性满足它。写成"按说话人标记切分；合并连续同人发言；按词频取每组
  前 3 词做标签；相邻组词汇重叠<0.2 处标记话题转换"（机械过程），BUILD 才有可实现
  的对象。
- **证据**：2026-07-04 08:30 运行——流水线首次端到端全通，唯一失败点即"抓第一句+幻觉
  标题+忽略第二说话人"，Critic 原文确认。
- **应对**：SPEC 模板要求算法深度为"程序员可逐字实现的分步过程"；validate_spec 机械
  拒绝无具体计算动词的描述。
- **推论**：对自治系统，规格的可执行性 > 规格的美感。检验标准：拿掉所有形容词后，
  规格还剩下可实现的操作吗？

## F-006 · 单文件工具不能承诺外部世界——模型会自然设计出需要语料库的方案然后造假

- **日期**：2026-07-03 · **状态**：确认
- **模型**：deepseek-reasoner(R1)（SPEC 设计侧）
- **声明**：不加约束时，SPEC 模型（R1）反复设计出需要"精选品牌语料库对照"、"行业基准
  数据库"、"预训练分类器"的工具方案。而工具实际是无数据库、无网络的单文件——BUILD
  无法诚实实现，只能造假，再被 Critic 拒绝。死循环。
- **证据**：2026-07-03 三次运行的 Critic 拒绝原因均包含"promises...curated corpus /
  no real algorithmic depth"。
- **应对**：validate_spec 自包含检查——算法深度/转换描述中出现 corpus / database of /
  pretrained 等承诺即拒绝，强制"只从用户输入本身计算"。
- **推论**：模型的设计先验来自见过的真实产品（有后端、有数据），部署约束必须作为
  硬闸写进验证器，写进 prompt 是不够的。

## F-007 · 把 HTML/CSS/JS 塞进 Python 字符串是稳定的语法错误源；raw 三引号 Jinja2 是解

- **日期**：2026-06-22 ~ 07-03 · **状态**：确认
- **模型**：deepseek-v4-flash（经 deepseek-chat 别名，6月）与 deepseek-v4-pro（07-03）均复现
- **声明**：模型生成"Python 里嵌 HTML"代码时，两类语法错误反复出现：① CSS/JS 反斜杠
  （`content:'\2014'`、正则 `/\d+/`）触发 "line continuation" 错误；② JS 大括号与
  f-string 冲突触发 "was never closed"。指定 `Template(r'''...''')`（raw + 三引号 +
  Jinja2）后此类错误消失。
- **证据**：2026-07-03 两次运行共 4 个语法错误全属此类；prompt 加 raw-string 强制规则后
  的运行零语法错误。
- **应对**：BUILD prompt 强制 raw 三引号 Jinja2 模板；重试反馈对这两类错误给专项诊断
  （区别于"代码太长被截断"）。
- **推论**：错误信息的正确分类是自治重试有效的前提——把转义错误误诊为截断，重试只会
  越改越短。

## F-008 · 自治流水线的失败会"上游化"——这是健康的调试轨迹

- **日期**：2026-06-19 ~ 07-04 · **状态**：观察中
- **模型**：跨越两代组合（06月 Gemini+deepseek-v4-flash；07-03 起 Qwen+DeepSeek R1/v4-pro），与单一模型无关
- **声明**：两周的失败位置持续向价值链上游移动：模型选错 → API 韧性 → 模式路由 →
  字符串工程 → 幻觉事实 → 规格质量。每修复一层，暴露的下一层失败都更接近"产品本身
  的难题"。当失败停在"输出不够聪明"时，管道工程阶段结束，产品阶段开始。
- **证据**：本文件 F-001 ~ F-007 的日期序列即该轨迹。
- **应对**：用"失败在哪一层"作为流水线成熟度的度量，而不是二元的成功率。
- **推论（待验证）**：当前架构下，工具智能度的天花板由"stdlib 单文件纯函数能表达的
  算法"决定。这个天花板与"创意专业人士愿意用第二次"的底线之间是否有交集，是本实验
  接下来要回答的核心问题。
- **2026-07-12 补充证据（状态不变，观察中）**：本周（07-06~07-12）14 次失败
  （07-10/07-11）没有延续"失败位置持续上移"的轨迹，反而退回了管道工程早期的低层
  缺陷——JS 语法未写完（`const b = p` 截断）、markdown 表格错位、DOM 不响应输入、
  照抄输入不转换（见 F-011）。这些工具的共同点是 SPEC 允许的输入格式范围比之前更宽
  （"Jira 工单、Figma 评论、客户邮件"三选一到"扫描 Figma 文件、PDF、ZIP 包"）。
  推测：F-008 的"上移"轨迹不是单向棘轮，而是与当次 SPEC 的范围收紧程度绑定——
  SPEC 一旦重新放宽，失败会退回到更早的层级。这个反例本身也支持 F-008 的核心应对
  （用"失败在哪一层"衡量成熟度），只是提示这个度量会随 SPEC 纪律松紧而波动，不会
  单调上升。
- **2026-07-19 补充证据（状态不变，观察中）**：本周（07-13~07-19）81 次尝试、
  3 次通过，失败原因 55 次 Critic 拒绝 / 26 次 Browser ground-truth 拒绝。
  07-14~16 三个连续 rest day 的 Critic 拒绝原文——"no real algorithmic depth,
  relying entirely on trivial regex"、"raw, truncated SVG code (cutting off
  mid-attribute)"、"basic SVG viewBox manipulation and DOM element selection
  boilerplate"——与 07-12 补充证据描述的低层缺陷（JS 语法未写完、DOM 不响应、
  照抄不转换）属同一族，不是新的失败类型。样本继续支持"非单调、随 SPEC 收紧
  程度波动"的推论：07-13/07-17（SPEC 范围相对收紧的环境型概念）各自在多次重试
  后于同一天内通过，07-14~16（Office Automation 的"brief 结构化""多版本冲突
  检测"类概念，范围更宽）连续三天在算法深度这一底层缺陷上全军覆没。样本量增加
  但推论未变，继续保持"观察中"。

## F-009 · 模型会发明条目把结构化输出"补完整"——形状完整性优先于内容诚实性

- **日期**：2026-07-04 · **状态**：观察中
- **模型**：deepseek-v4-pro（生成）；qwen3.7-max（识破）
- **声明**：当输出是结构化列表/时间轴/表格时，若输入内容不足以填满"预期的形状"，
  模型倾向于**发明条目补位**而不是输出诚实的短列表。例：时间轴章节工具在输入无结尾
  信息时，编造了一个 01:10:00 时间码的"Conclusion"章节。与 F-004（编造外部事实）
  不同，这是**编造输入内该有而没有的内容**——对"完整形状"的先验压倒了对输入的忠实。
- **证据**：2026-07-04 14:30 运行，Critic 拒绝原因原文："hallucinating a generic
  'Conclusion' chapter at an arbitrary 01:10:00 timecode to finish the timeline"。
  同日 08:30 运行的"忽略第二说话人"是同族问题的另一面（该省不省、该有没有）。
- **应对**：BUILD prompt 加可溯源规则（每个输出条目必须能追溯到输入的具体位置；
  3 个真实条目优于 5 个含伪造的）；重试反馈新增"填充物"专项分支——明确这是删除
  任务而非补充任务。
- **推论**：结构化输出的诚实性需要显式约束"输出长度跟随输入内容"，否则模型会按
  训练数据里"这类表格通常长什么样"来补形状。

## F-010 · 约束的跷跷板效应——禁止伪造后，模型摆向"诚实地拒绝工作"

- **日期**：2026-07-04 · **状态**：观察中
- **模型**：deepseek-v4-pro（生成侧）
- **声明**：对同一深层行为（对输入不忠实）从一侧加约束，模型会摆向对侧失败而不是
  收敛到中间态。时间线：14:30 模型**伪造**条目补全输出 → 部署"禁止伪造、每条目可溯源"
  规则 → 20:30 模型不伪造了，但输入缺少说话人标记时**直接拒绝**（返回一行
  "No speaker-turn segments found. Please check the format."）。两种失败共享同一根因：
  模型缺少"优雅降级"的中间概念——主要结构标记缺失时退回粗粒度切分（段落→句子→定长块）
  继续尽力工作。
- **证据**：2026-07-04 三次运行日志的行为序列（伪造 → 填充 → 拒绝），其中 20:30 的
  拒绝发生在反伪造规则部署之后 6 小时。
- **应对**：BUILD prompt 明确"优雅降级"为强制要求（主标记缺失→逐级退回粗粒度切分，
  限制说明只能附注在真实输出后，不能替代输出）；rigid-input 重试分支扩展关键词，
  接住"礼貌拒绝"类输出。
- **推论**：给生成模型定行为边界时，禁止性规则必须配对指明"正确的中间态是什么"，
  否则模型在被禁行为的对立面找最近的出口。单侧围堵会制造钟摆。

## F-011 · 输出结构给定但转换步骤未写成机械过程时，模型退化为"复制粘贴"而非转换

- **日期**：2026-07-10 · **状态**：确认
- **模型**：deepseek-v4-pro（BUILD）；qwen3.7-max（识破）
- **声明**：SPEC 要求"把原始项目材料（Jira 工单/Figma 评论/客户邮件/设计时间线）转换
  为结构化叙事、时间线或摘要"，但算法深度没有写成 F-005 要求的机械步骤时，BUILD 会
  稳定产出"输出结构照抄输入结构"的假转换——把原始文本块整段搬进输出模板的字段里，
  不做任何提炼、判断或重组。这与 F-004（编造事实）、F-009（编造条目补形状）是同一
  失败家族的第三种变体：这次模型连编都不编，直接照搬。
- **证据**：2026-07-10 同一天内三个独立工具概念（"Jira/Figma/客户邮件案例研究生成器"
  "按时间顺序的设计产物工具""raw project notes 转换工具"）被 Critic 以近乎相同措辞
  拒绝——"output structure is identical to the input structure with no real
  transformation"、"simply copy-pasting the entire raw text blob"、"blindly
  copy-pastes the raw input text verbatim"。三个不同工具概念的独立复现，不是同一工具
  的重试。
- **应对**：BUILD prompt 增加 passthrough 自检规则——产出前比对输出各字段与输入原文
  的逐字重叠度，某字段与输入子串重叠 >70% 判定为未转换，必须重写而非直接引用；
  SPEC 阶段要求"转换"类工具必须写出至少一个输入到输出的判别式操作（排序/筛选/打分/
  归类/重写），不能只写"结构化"这类描述性词。
- **推论**：F-005 揭示的"愿景式规格导致偷懒实现"在这里的具体形态是偷懒到了底线——
  算法深度描述不够机械时，模型选择的最省力路径不是编造（F-004/F-009），而是原样
  照搬输入。编造、补形状、照搬这三种偷懒模式共享同一根因：SPEC 对"转换"动作定义得
  不够可执行，模型就会退回到它能确定不出错的操作——什么都不做。

## F-012 · Ground-truth 拒绝后的重试会换壳但不换病灶——同一失败指纹在同一工具的连续尝试中原样复现

- **日期**：2026-07-11 · **状态**：观察中
- **模型**：deepseek-v4-pro（BUILD，Mode 3）；Playwright 无头浏览器地面真值门（非 LLM，见 F-001）
- **声明**：Mode 3 工具被 Playwright 地面真值拒绝（"DOM did not react to input"）后，
  BUILD 收到的重试反馈只有症状级数据（fields_filled / text_changed / 节点数），不含
  "哪一行 JS 没有绑定事件监听器"或"处理函数没有写回 DOM"这类机制级信息。结果：同一
  工具连续 4 次重试，DOM 节点总数从 34 变为 52（说明代码结构确实被重写，不是简单
  复制），但核心病灶（`text_changed` 恒为 `False`）原样复现了 4 次。
- **证据**：质量账本 2026-07-11 同一工具连续 4 条记录：
  `fields_filled=4, text_changed=False, nodes 34->34`（×3）、
  `fields_filled=4, text_changed=False, nodes 52->52`（×1）。
- **应对**：Playwright 探测器需要从"前后节点数对比"升级为机制级诊断——例如检测目标
  input 元素是否绑定了 change/input 监听器，若绑定了，监听器函数体是否实际写入了
  目标节点的 textContent/innerHTML；重试反馈应把这条诊断原文喂给 BUILD，而不是复述
  "DOM 没反应"这一症状描述。
- **推论**：地面真值门解决了"要不要相信模型自称做到了"（F-001），但还没解决"模型
  如何修复它做错的具体机制"——症状级反馈只能触发换壳式重写，机制级反馈才能触发
  针对性修复。这是 F-001 之后地面真值系统需要补的下一层，样本量仅 1 个工具、4 次
  尝试，需要更多独立事件才能从"观察中"升级为"确认"。
## F-013 · 自我修改流水线的写入路径需要与外部编辑同等的语法防护，两者不会自动共享

- **日期**：2026-07-12（事故）/ 2026-07-17（修复）· **状态**：确认
- **模型**：deepseek-v4-pro（周进化文本生成，写入 `lili_engineering.py` /
  `lili_blindspot.py` / `lili_soul.py` 三个 carrier 文件）
- **声明**：项目已有 PostToolUse hook 在 Claude 编辑 `src/*.py` 后自动跑
  `ast.parse()` 校验语法（见 CLAUDE.md Phase 8），但周进化管道自己把模型生成的
  反思文本（工程教训示例、盲点分析）拼进同一批 `.py` 文件的三引号字符串字面量时，
  完全没有等价校验。2026-07-12 一次运行生成的工程教训示例里包含字面
  `"""..."""` docstring，提前闭合了外层三引号字符串，导致两个文件从那次提交起
  持续 `SyntaxError`，直到次日跑测试才被发现（`test_prompts.py` 的 import 链
  `lili_prompts` → `lili_blindspot` 报错）。防护存在于"Claude 改代码"这条路径，
  不存在于"Lili 自己改代码"这条路径——两条路径共享同一份被写入的文件，却不共享
  同一份防护，是因为防护被实现为外部工具（hook）而不是写入函数本身的前置条件。
- **证据**：CLAUDE.md Pitfalls Logged 对应条目；提交 `0955e35` 的
  `_sanitize_embedded()`（转义三引号与结尾反斜杠）与 `_guarded_write()`
  （写入前 `ast.parse()` + `exec()` + 必需符号名检查，任一失败则保留上周文件、
  拒绝写入）；提交信息记录"verified end-to-end with the exact 07-12 killer
  input"，即用引发事故的原始 payload 重放，确认新代码路径下文件仍能正确解析、
  导入、内容保留；新增 10 个单测覆盖净化器与 guarded-write 的拒绝/接受矩阵。
- **应对**：任何"系统改写自己代码"的写入函数，防护必须内建在写入函数本身
  （写入前双重校验：语法 + 语义/符号完整性），不能依赖外部编辑钩子——因为自我
  修改的调用路径天然不经过那个钩子。
- **推论**：F-007 指出"把 HTML/JS 塞进 Python 字符串是稳定的语法错误源"，这条
  发现是它的自指版本——当"塞进字符串"的内容来自系统自己的反思输出而不是外部
  代码时，同一类错误一样会发生，且更危险，因为没有人在旁边跑 hook。自治系统里
  "写入自己源码"的每一处，都应该被当成需要 ground truth 校验的边界，而不是
  信任内部调用。

## F-014 · 重试到成功为止的逻辑对"本质上不可能"的任务没有退出条件——会无限期消耗主循环

- **日期**：2026-07-14 ~ 07-16（事故）/ 07-16（修复）· **状态**：确认
- **模型**：与具体生成模型无关——deepseek-reasoner(R1)/deepseek-v4-pro 在 SPEC/BUILD
  阶段反复尝试同一 Issue，问题不在模型输出质量，在管道的终止条件设计
- **声明**：Issue 委托流程（见 CLAUDE.md Issue Commission Flow）原先的逻辑是
  "每天重试，直到出现 `lili-built` 标签"，对可行任务是合理的重试语义，但对
  **本质上超出单文件沙盒能力**的任务（Issue #5：整理手机截图，需要视觉 AI +
  OCR，而工具运行环境是无网络、无后端的单文件 HTML/JS）等价于死循环——因为
  `lili-built` 永远不会出现，管道会一直把每天的构建配额耗在同一个不可能任务上，
  而不会退回到自由选题。
- **证据**：质量账本"repeat-offender concepts"统计显示该截图整理概念失败 9 次
  （`lili_ledger_report.py` 输出）；日记技术备注确认 2026-07-14~16 三个连续
  rest day 均源于同一委托的重试；提交 `5d0091e` 标题明确点出根因
  "Issue #5 ... consumed 3 consecutive days (07-14~16) because commissions
  retry daily until lili-built, forever"。
- **应对**：新增 `lili-blocked` 标签与两级失败响应——第一次失败发诚实的进度
  说明评论，第二次失败发能力边界说明并打上 `lili-blocked`，之后
  `fetch_tool_requests` 跳过被拦标签的 Issue，管道当天回到自由选题；Issue 本身
  保持打开（暂停而非放弃，未来能力扩展后可重新评估），不自动关闭。
- **推论**：这与 F-006（单文件工具不能承诺外部世界）是同一类"部署约束必须写进
  硬闸"问题的另一面——F-006 管的是"模型设计阶段会不会许下做不到的承诺"，这条
  管的是"当承诺已经许下且不可能兑现时，管道要不要有止损机制"。自治系统里任何
  "重试直到成功"的循环都需要显式的失败计数上限和退出路径，否则任务是否可行
  这件事只能靠管道被目睹"卡住"来发现，而不是被设计出来预防。

## F-015 · 重试反馈的关键词路由存在自我遮蔽——更具体的诊断分支被更早、更宽泛的分支吃掉

- **日期**：2026-07-04（引入）~ 07-24（发现）· **状态**：确认，已修复
- **模型**：deepseek-v4-pro（生成）；qwen3.7-max（Critic，产出了触发遮蔽的措辞）
- **声明**：F-009 的"填充物"专项重试分支（07-04 加入）用关键词 `hallucinat/invented/
  padded` 匹配，但代码顺序上排在更早、更宽泛的"generic/static"分支**之后**。Critic
  给出的真实拒绝理由经常同时包含"generic"（如"a generic 'Conclusion' chapter"）和
  "hallucinat"，于是被前一个分支截获，从未走到为它专门写的、更具体的"删除任务"反馈。
  这个 bug 隐藏了 20 天，因为两个分支的反馈都"合理"，日志里看不出路由错了。
- **证据**：`tests/test_retry_routing.py` 新增的路由回归测试首次运行即失败
  （`test_padding_not_shadowed_by_identical`）；用真实账本核对，07-04/07-06/07-10/
  07-16 共 4 条应属 padding 类的失败记录，原因文本同时含"generic"和"hallucinat"，
  证实生产环境确实一直在走错分支。同一天（07-24）的浏览器验证连续 4 次拒绝同一
  "字距微调"工具、原因逐字重复，是这类"反馈不具体→模型无法收敛"问题的独立佐证。
- **应对**：把 padding 分支移到 generic/static 分支**之前**（更具体的诊断应优先于
  更宽泛的）；新增 `tests/test_retry_routing.py` 把整条 elif 链的匹配顺序做成
  独立、带真实历史用例的回归测试，而不是每次改动后手动跑一次性验证脚本。同时给
  浏览器地面真值探针加上控制台报错捕获（`console_errors=`），让"DOM 没反应"这类
  失败的重试反馈从"泛泛建议"变成"具体检查这几个绑定点"。
- **推论**：字符串关键词路由的分支顺序是隐性契约，且极易被后续新增分支破坏而不报错
  ——每次新增 elif 分支都必须验证它是否会被更早的分支吃掉，也可能反过来吃掉更晚的
  分支。这类系统的正确形态是显式优先级表 + 回归测试，而不是新增代码时凭直觉插入
  一个 elif。
